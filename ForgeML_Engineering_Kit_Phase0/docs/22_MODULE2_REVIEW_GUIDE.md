# Module 2 — Review Guide

**Do not read every file.** Module 2 is ~1,100 lines of source and ~1,400 of
tests. About 20% of it carries essentially all of the engineering risk, and this
document points you at exactly that.

**Total review time: ~75 minutes.** If you only have 25, read §1 and §2 and stop.

---

## §1 — The highest-risk code (25 min)

### `infrastructure/database/repositories.py` — read in full (15 min)

This is where every race lives.

| Function | Why it matters |
| --- | --- |
| `SqlAlchemyPackageCatalog.get_or_create` | Two identical uploads race. Correctness depends on the `sha256` unique index, **not** on the `_by_checksum` check above it. The `except IntegrityError` savepoint is the code that actually runs under contention. **If this is wrong, concurrent uploads 500.** |
| `SqlAlchemyOperationStore.begin` | Same shape, higher stakes: if the race produced two operations, **the same work would execute twice** — an idempotency key exists precisely to prevent that. |
| `SqlAlchemyOperationStore.claim_next` | `FOR UPDATE SKIP LOCKED`. The whole ADR-010 worker guarantee. Note the `types is not None` check — an empty lane must select *nothing*, and truthiness would make it select *everything*. |
| `SqlAlchemyOperationStore.recover_orphaned` | ADR-016. Without it, a killed worker strands an operation in `RUNNING` forever. |
| `save_validation` | The only place a package changes state. Note `manifest_version` is set here and nowhere else. |

### `infrastructure/database/unit_of_work.py` — read in full (5 min)

25 lines. `__exit__` always rolls back; only an explicit `commit()` persists. If
this is wrong, **an audit event could be committed for a state change that was
not** — which would make the audit trail fiction (docs 04).

### `migrations/versions/0d7adf1f94cf_*.py` — read the trigger DDL only (5 min)

`IMMUTABILITY_DDL`. Three plpgsql triggers enforce ADR-003 immutability and
docs 04 append-only audit *at the database*, against an operator with `psql`.
Check the logic; the surrounding `create_table` calls are autogenerated and
verified by an anti-drift test.

## §2 — The tests that decide whether you can trust the above (20 min)

Read the test **names** — they are the specification. Read the bodies of these:

| Test | What it proves |
| --- | --- |
| `test_insert_races.py` (both) | The races in §1 genuinely occur and resolve to one row. These force both transactions past their existence check with a barrier — the sequential tests never reach the `IntegrityError` branch at all. |
| `test_concurrency.py::test_two_workers_never_claim_the_same_operation` | Two live transactions, `SKIP LOCKED` works. |
| `test_port_conformance.py::test_an_exception_mid_transaction_leaves_nothing_behind` | Rollback is real. |
| `test_invariants.py` | The triggers hold when the repositories are bypassed. |
| `test_migrations.py::test_the_models_match_the_migration` | Empty autogenerate diff — catches a model changed without a migration. |

## §4 — Questions you should answer (15 min)

1. **`operations.target_id` is text with no foreign key.** An operation targets a
   package (sha256) or, from phase 5, a version (uuid) — no single FK spans both.
   Referential integrity therefore lives in the repository, not the database.
   **Do you accept that, or do you want a `targets` supertable before phase 5
   makes it expensive to change?** *(This is the schema's weakest joint.)*

2. **`MAX_ATTEMPTS = 3`, and there is no automatic retry.** A crashed worker
   re-queues an operation up to three times, then abandons it. **Is 3 right, and
   do you want operator-visible alerting when an operation is abandoned?**

3. **`recover_orphaned` heals only at startup.** A worker that *hangs* without
   dying holds its row forever. Acceptable under one supervised worker; wrong the
   day you run two. **Confirm the single-worker cap is real.**

4. **Findings are JSONB, not a queryable table.** Confirm phase 9 will never need
   "show me all packages failing X".

5. **`validator_version` is a hand-typed string.** A validator bugfix that does
   not bump it keeps the stale verdict. **Should it derive from the package
   version instead?**

## §5 — Implementation risks

| Risk | Severity | Mitigation in place |
| --- | --- | --- |
| A race handler is wrong and only fires in production | High | Forced race tests; both branches covered |
| A model changes without a migration | High | Anti-drift test in CI |
| An ORM object escapes infrastructure and lazy-loads | Medium | AST test forbids `sqlalchemy` outside `infrastructure/database/`; mappers rebuild every value as a frozen record |
| The fakes drift from PostgreSQL and mislead Module 3 | Medium | One conformance suite runs against both — it has already caught two divergences |
| A hung (not dead) worker strands an operation | Medium | Accepted; see question 3 |
| Trigger DDL is wrong and silently permits mutation | Medium | `test_invariants.py` bypasses the repositories to check |

---

## What I would look at first if I had 10 minutes

`get_or_create` and `begin` in `repositories.py`, then
`test_insert_races.py`. If those four are right, Module 2 is sound. If they are
wrong, nothing else matters.
