## Release description

> ForgeML 0.9.1 is the engineering baseline. It adds no features.
>
> ForgeML is a single-server control plane for deploying machine-learning models
> as isolated, versioned HTTP services. You upload a `.forge` package, it is
> validated and analyzed, a container image is built deterministically, a
> hardened container is started, and one version at a time serves predictions at
> a stable name. Activation and rollback are atomic. Every operation is durable
> and survives a restart.
>
> This release exists to make that description verifiable rather than merely
> claimed. Everything the repository asserts about itself was re-derived from a
> clean clone: the test count, the coverage number, the build, the wheel, the
> locks, and the determinism of the package format. One security upgrade and
> three consistency fixes came out of that exercise, and they are the whole
> changelog.
>
> **Pre-1.0: no compatibility guarantee.** The `/v1` API, the `.forge` format,
> and the Python distribution version independently; the compatibility promise
> in `docs/RELEASE.md` takes effect at 1.0.

## Engineering highlights

**Security.** `python-multipart` upgraded 0.0.20 → 0.0.32, clearing six
advisories. Five were reachable: they sit on the multipart parser behind package
upload, which has no authentication in front of it. Two denial-of-service paths,
an unbounded read on a negative `Content-Length`, and a urlencoded parser
differential against the WHATWG standard. Both lock files now audit clean.

**A concurrency guarantee that was claimed but never proven.** `ActivationService`
documents that the route swap is atomic under the deployment row lock. The lock
is plain `FOR UPDATE`, not `SKIP LOCKED`, precisely so a second activation waits
rather than proceeding in parallel — but no test drove two callers at the same
deployment. There is one now, against real PostgreSQL, and it was verified to
fail when the lock is weakened. The invariant is enforced rather than described.

**The version was wrong on the wire.** `/health` and `/ready` report the installed
distribution version, which was still the `0.1.0` scaffold default while every
document said 0.9.

**Reproducibility, checked rather than assumed.** A clean clone builds, tests,
packages, and smoke-tests without a single undocumented step. Both lock files
recompile byte-identically. The example `.forge` package rebuilds to the same
SHA-256, which is the determinism claim in ADR-003 holding in practice.

## Compatibility statement

ForgeML is `0.x` and carries **no compatibility guarantee**. Anything may change
before 1.0, including the `/v1` surface and the `.forge` format. The
compatibility promise in `docs/RELEASE.md` — no endpoint removal, no field
removal, no error-code meaning change — takes effect at 1.0, which requires
Modules 8, 9, and 10 complete and frozen.

Supported: the latest release only.

```bash
git clone https://github.com/Riicko-19/ForgeML && cd ForgeML
make setup && make db && make migrate && make run
```

Requires Python 3.11, Docker, and PostgreSQL 16 (`make db` provides the last).

## Known limitations

Stated plainly because a baseline that hides its gaps is not a baseline:

- **No authentication or authorization.** The control plane is unauthenticated.
  Do not expose it to an untrusted network.
- **The control plane is root-equivalent** through the Docker daemon, by design.
  Anyone who can reach it can run containers on the host.
- **Packages are trusted by design.** Deploying a `.forge` package means running
  its code. Validation checks structure, not intent.
- **Build-time execution is less isolated than runtime.** Runtime containers are
  hardened per ADR-001; the build step is not hardened to the same degree.
  Documented in `SECURITY.md`.
- **Artifacts accumulate without bound.** ADR-012's retention policy is recorded
  but not implemented.
- **Docker tests skip silently.** They ran for this release, but a run on a
  machine without a daemon reports green while proving nothing about Module 6.
- **No monitoring, metrics, or rate limiting.** Module 8 and later.
- **Single server, single worker.** Not horizontally scalable, on purpose.

## Deferred engineering decisions

- **ADR-022 — where authentication belongs.** The frozen roadmap defines no
  authentication phase. ADR-018, ADR-019, and ADR-020 prepared for it, but
  assigning it a phase amends the roadmap and requires its own ADR. This is the
  decision blocking the next module.
- **ADR-012 retention policy** — recorded, unimplemented.
- **Four deferred API items** under ADR-020, including
  `GET /v1/deployments/{id}/versions`, held back so they arrive authorized
  rather than being retrofitted.
